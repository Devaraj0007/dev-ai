# Research Agent (with Citations)

A CLI research agent that answers questions **strictly from a set of provided
source documents**, citing which source every claim came from, and explicitly
saying when the sources don't have the answer.

Built for the 24-Hour AI Agent Challenge (Research Agent with Citations,
Advanced track).

---

## What it does

1. **Ingest** — reads `.txt`, `.md`, or `.pdf` files from `data/sources/`,
   splits them into citation-sized chunks, and builds a TF-IDF retrieval
   index (`src/ingest.py`).
2. **Retrieve** — for a given question, finds the most relevant chunks by
   cosine similarity (`src/retriever.py`). Chunks below a similarity
   threshold are dropped rather than force-fed to the model.
3. **Synthesize with citations** — sends the retrieved chunks (numbered) and
   the question to Claude, with a system prompt that requires every claim to
   carry a `[n]` citation marker and forbids using outside knowledge
   (`src/agent.py`).
4. **Validate grounding** — after the model responds, the agent checks that
   every citation number used actually corresponds to a chunk that was
   really retrieved. If the model invents a citation number, the response is
   flagged `grounded: false` rather than silently trusted.

This is the same retrieval-augmented-generation pattern used in my earlier
project *Ask My Docs* (LangChain + FAISS + GPT-4o-mini), rebuilt here with a
lighter, fully-offline retrieval layer so the whole thing installs and runs
in minutes with zero external services besides the Claude API.

## Setup

```bash
git clone <this-repo-url>
cd research_agent
python3 -m venv .venv && source .venv/bin/activate   # optional but recommended
pip install -r requirements.txt

cp .env.example .env
# then edit .env and set ANTHROPIC_API_KEY=sk-ant-...
```

Requires Python 3.9+. No GPU, no vector DB server, no Docker — everything
runs locally except the single call to the Claude API per question.

---

## Running it

**1. Build the index** (from the sample sources, or point `--source-dir` at
your own folder of `.txt` / `.md` / `.pdf` files):

```bash
python -m src.cli ingest
```

**2. Ask a single question:**

```bash
python -m src.cli ask "How many weeks of paid parental leave do employees get?"
```

**3. Run the full sample question set and save results:**

```bash
python -m src.cli batch data/questions.txt --out outputs/sample_qa.json
```

**4. Interactive mode:**

```bash
python -m src.cli interactive
```

**5. Check citation/grounding metrics on a batch run:**

```bash
python eval.py outputs/sample_qa.json
```

## Sample output

For the question *"How many weeks of paid parental leave do employees get
once they've completed 12 months of service?"*, the agent retrieves the
`company_handbook.txt` chunk on parental leave and returns something like:

```json
{
  "question": "How many weeks of paid parental leave do employees get once they've completed 12 months of service?",
  "answer": "Employees who have completed at least 12 months of continuous service are eligible for 16 weeks of fully paid parental leave [1].",
  "sources_sufficient": true,
  "gap_note": "",
  "grounded": true,
  "citations": [
    {"marker": "[1]", "source": "company_handbook.txt", "excerpt": "3. Parental Leave\nNorthwind Analytics offers 16 weeks..."}
  ]
}
```

For an out-of-scope question like *"Who is Northwind Analytics' current
CEO?"*, no source document mentions a CEO, so retrieval either returns
nothing above the similarity threshold or returns weakly-related chunks, and
the agent is instructed to set `sources_sufficient: false` and explain the
gap rather than guessing.

The full set of 10 sample Q&A pairs (8 answerable across all three source
documents, 2 deliberately out-of-scope) lives in `outputs/sample_qa.json`
after running the `batch` command.

## Design decisions & tradeoffs

**Retrieval: TF-IDF + cosine similarity, not neural embeddings.**
I chose scikit-learn's TF-IDF over a sentence-transformer embedding model on
purpose. TF-IDF is lexical (keyword-overlap) rather than semantic, so it
will miss paraphrased or synonym-heavy queries (e.g. it matches "parental
leave" fine, but wouldn't strongly match "time off for a new baby" against
the same chunk). In exchange, it has zero model-download weight, is fully
deterministic, runs instantly on CPU, and is easy to explain and defend in
an interview. With more time, I'd add a hybrid retriever (TF-IDF/BM25 for
exact terms + a small embedding model for semantic recall) and compare
recall on a held-out question set.

**Chunking: paragraph-first, sentence-level fallback.**
Source documents here are already well-structured (numbered sections), so
paragraph-level chunks map cleanly to a single citable idea. Long paragraphs
fall back to sentence-window chunking with a 1-sentence overlap so a chunk
boundary doesn't silently cut a fact in half. This is simple and works well
for structured docs; it would need tuning (e.g. semantic chunking) for
messier source material like raw OCR output.

**Grounding validation is a hard check, not a prompt request.**
Asking the model nicely not to invent citations isn't enough — I added a
post-hoc validator (`_validate_grounding` in `agent.py`) that parses every
`[n]` marker out of the answer and confirms `n` is within the range of
chunks that were actually retrieved for that question. If the model cites a
number that doesn't exist, the response is flagged `grounded: false` instead
of being trusted. This is the single most important reliability feature in
the project — it's the difference between "the model was told to cite
correctly" and "the citations are verified to be real."

**Structured JSON output over free-text parsing.**
The model is asked to return a single JSON object (answer, sufficiency
flag, gap note) rather than free text, which makes `sources_sufficient` a
first-class, machine-checkable field instead of something I'd have to guess
at by string-matching phrases like "I don't know." There's a regex fallback
in `_parse_model_json` for the rare case the model wraps the JSON in
markdown fences or adds stray text.

**What I'd improve with more time:**
- Add the hybrid/semantic retrieval mentioned above and measure recall@k on
  a labeled question set.
- Add a second LLM pass (or a smaller/cheaper model) as an independent
  "faithfulness checker" that verifies each citation's excerpt actually
  supports the claim next to it, rather than only checking that the
  citation number is in range.
- Support multi-turn follow-up questions with conversation memory.
- Add a lightweight web UI (the CLI is deliberately the fastest path to a
  working, testable agent within the time budget).

## Known limitations

- Retrieval is lexical, so heavily paraphrased questions may under-retrieve
  (see tradeoffs above).
- The grounding validator checks that citation *numbers* are real — it does
  not (yet) verify that the cited excerpt actually semantically supports the
  specific claim next to it. That's the "faithfulness checker" noted above.
- PDF extraction quality depends on `pypdf`, which struggles with scanned
  (image-only) PDFs; those would need OCR first.
