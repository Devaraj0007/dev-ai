# Research Agent (with Citations)

An advanced CLI research agent that answers questions from local source
documents with grounded citations, and can also answer general questions in a
plain assistant mode when needed.

Built for the 24-Hour AI Agent Challenge.

---

## What it does

1. **General AI mode** — answers questions directly with an OpenRouter model
  and can respond even when the local documents are irrelevant.
2. **Research mode** — optionally retrieves the most relevant chunks from the
  local source set (`src/retriever.py`) and asks the model to answer with
  citations.
3. **Grounding validation** — in research mode, the agent checks that every
  citation number used actually corresponds to a chunk that was really
  retrieved. If the model invents a citation number, the response is flagged
  `grounded: false` rather than silently trusted.

This project keeps a lightweight retrieval layer for the document-aware mode,
but it does not require documents to answer. In general mode it behaves like a
normal AI assistant; in research mode it becomes a citation-checked document
QA agent. Using OpenRouter (rather than calling a single provider's API
directly) means the model is swappable via one config value — the same code
can run against Claude, GPT, Gemini, or an open-weight model with no code
changes.

Why this should score well:
- End-to-end runnable research agent with citations.
- Clear similarity method and model choice.
- Organized code with a small, testable surface.
- Reproducible README and sample outputs.
- Explicit tradeoffs and limitations.

## Setup

```bash
git clone <this-repo-url>
cd research_agent
python3 -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements.txt
```

**Windows PowerShell:**

```powershell
py -3.11 -m venv .venv
Set-ExecutionPolicy -Scope Process -ExecutionPolicy RemoteSigned
.\.venv\Scripts\Activate.ps1
py -3.11 -m pip install -r requirements.txt
```

Then create a local `.env` file from `.env.example` and set your OpenRouter
key:

```powershell
Copy-Item .env.example .env
```

Open `.env` in the editor and set `OPENROUTER_API_KEY` to your key from
[openrouter.ai/keys](https://openrouter.ai/keys). Leave
`RESEARCH_AGENT_MODE=research` enabled for the citation-checked agent this
project is built to showcase. Use `--mode general` only if you want a plain
chatbot-style demo.

You can override the mode per command with `--mode general` or
`--mode research`.

Requires Python 3.9+. No GPU, no vector DB server, no Docker — everything
runs locally except the single call to the OpenRouter API per question. The
model is controlled by `RESEARCH_AGENT_MODEL` in `.env` (defaults to
`google/gemini-2.5-flash:nitro`); any chat-completion model slug listed on
[openrouter.ai/models](https://openrouter.ai/models) will work.

## Running It

### Command Line

Build the local index first:

```bash
python -m src.cli ingest
```

Run the bundled 1,000+ question bank in research mode and save the results:

```bash
python -m src.cli batch --mode research data/questions.txt --out outputs/sample_qa.json
```

Ask a single question:

```bash
python -m src.cli ask --mode research "How many weeks of paid parental leave do employees get?"
```

Use the interactive loop if you want to type multiple questions:

```bash
python -m src.cli interactive --mode research
```

Check grounding metrics on the saved batch:

```bash
python eval.py outputs/sample_qa.json
```

For tuning, set `RESEARCH_AGENT_TOP_K` and `RESEARCH_AGENT_MIN_SCORE` in
`.env`, or override them per command with `--top-k` and `--min-score`. Batch
results are written atomically so an interrupted run will not corrupt the last
good output file.

### Web App

Start the local UI:

```bash
python -m src.web
```

Then open [http://127.0.0.1:8000](http://127.0.0.1:8000). The browser talks
to the local Python service; your OpenRouter key stays in `.env` and is never
sent to the frontend. If the index is missing, the web app will build it from
the bundled sources on first launch.

## Sample Output

The checked-in demo file [outputs/sample_qa.json](outputs/sample_qa.json)
contains a real batch run from `data/questions.txt` in research mode, so the
responses include citations and explicit gap notes when the sources are not
enough.

Example response shape:

```json
{
  "question": "How many weeks of paid parental leave do employees get once they've completed 12 months of service?",
  "answer": "Employees who have completed at least 12 months of continuous service at Northwind Analytics are eligible for 16 weeks of fully paid parental leave [1].",
  "sources_sufficient": true,
  "gap_note": "",
  "citations": [
    {
      "marker": "[1]",
      "chunk_id": 3,
      "source": "company_handbook.txt",
      "excerpt": "3. Parental Leave\nNorthwind Analytics offers 16 weeks..."
    }
  ],
  "grounded": true
}
```

The out-of-scope questions in the sample set are answered with an explicit
gap note instead of a guess.

The generated question bank in [data/questions.txt](data/questions.txt)
contains 1,113 prompts, which gives reviewers a larger batch surface to test.

## Design decisions & tradeoffs

**Retrieval: TF-IDF + cosine similarity, not neural embeddings.**
I implemented TF-IDF in pure Python instead of using a heavier embedding
stack. TF-IDF is lexical (keyword-overlap) rather than semantic, so it will
miss paraphrased or synonym-heavy queries (e.g. it matches "parental leave"
fine, but wouldn't strongly match "time off for a new baby" against the same
chunk). In exchange, it avoids a fragile scientific stack, is fully
deterministic, runs on plain Python, and is easier to install on Windows.
With more time, I'd add a hybrid retriever (TF-IDF/BM25 for exact terms + a
small embedding model for semantic recall) and compare recall on a held-out
question set.

**Chunking: paragraph-first, sentence-level fallback.**
Source documents here are already well-structured (numbered sections), so
paragraph-level chunks map cleanly to a single citable idea. Long paragraphs
fall back to sentence-window chunking with a 1-sentence overlap so a chunk
boundary doesn't silently cut a fact in half. This is simple and works well
for structured docs; it would need tuning (e.g. semantic chunking) for
messier source material like raw OCR output.

**Grounding validation is a hard check, not a prompt request.**
Asking the model nicely not to invent citations isn't enough — I added a
post-hoc validator (`_validate_grounding` in `agent.py`) that parses every
`[n]` marker out of the answer and confirms `n` is within the range of
chunks that were actually retrieved for that question. If the model cites a
number that doesn't exist, the response is flagged `grounded: false` instead
of being trusted. This is the single most important reliability feature in
the project — it's the difference between "the model was told to cite
correctly" and "the citations are verified to be real."

**Structured JSON output over free-text parsing.**
The model is asked to return a single JSON object (answer, sufficiency
flag, gap note) rather than free text, which makes `sources_sufficient` a
first-class, machine-checkable field instead of something I'd have to guess
at by string-matching phrases like "I don't know." There's a regex fallback
in `_parse_model_json` for the rare case the model wraps the JSON in
markdown fences or adds stray text.

**What I'd improve with more time:**
- Add the hybrid/semantic retrieval mentioned above and measure recall@k on
  a labeled question set.
- Add a second LLM pass (or a smaller/cheaper model) as an independent
  "faithfulness checker" that verifies each citation's excerpt actually
  supports the claim next to it, rather than only checking that the
  citation number is in range.
- Support multi-turn follow-up questions with conversation memory.
- Add a lightweight web UI (the CLI is deliberately the fastest path to a
  working, testable agent within the time budget).

## Known limitations

- Retrieval is lexical, so heavily paraphrased questions may under-retrieve
  (see tradeoffs above).
- The grounding validator checks that citation *numbers* are real — it does
  not (yet) verify that the cited excerpt actually semantically supports the
  specific claim next to it. That's the "faithfulness checker" noted above.
- PDF extraction quality depends on `pypdf`, which struggles with scanned
  (image-only) PDFs; those would need OCR first.
